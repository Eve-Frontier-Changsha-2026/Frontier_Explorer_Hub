# Frontier Explorer Hub - 情境測試 (Scenario Tests)

本文件定義了 Frontier Explorer Hub 的核心應用情境測試，用於驗證「隱私保護熱力圖」、「情報訂閱系統」及「情報懸賞」等機制的邏輯正確性與世界觀契合度。

## 情境一：熱力圖訂閱分級測試 (Subscription Tier Verification)
**目的**：確保不同付費等級的玩家，所接收到的熱力圖情報解析度與實效性符合設計。

- **前置條件**：
  1. 星系 A01 最近 5 分鐘內發生大規模戰鬥（產生 50 筆 Killmail）。
  2. 玩家 X 為「免費方案 (Free Tier)」用戶。
  3. 玩家 Y 已支付 SUI 訂閱「高級方案 (Premium Tier)」。
- **前置條件實作 (Data Sourcing)**：
  - **讀取來源**：呼叫 EVE EYES API (`GET /api/indexer/move-calls`)，過濾 `moduleName=turret`（防禦/交戰）與 `assembly`（製造/消耗）。
  - **技術對接**：透過背景 Cron Job 定期獲取這些 module 的操作次數（Move Calls），並從 payload 解析出位置資料；接著將活躍度加總寫入 `region_enrichment` 資料表，成為支撐熱力圖的量化指標。
- **測試步驟**：
  1. 玩家 X 打開 Frontier Explorer Hub 查看星系 A01。
  2. 玩家 Y 打開 Frontier Explorer Hub 查看星系 A01。
- **預期結果**：
  1. 玩家 X 的星圖上，星系 A01 未顯示任何異常（因為熱力圖具備 30 分鐘延遲）。
  2. 玩家 Y 的星圖上，星系 A01 立即亮起高危險紅色警告，並顯示「高戰鬥值異常 / 預估艦隊規模：大」。
  3. 30 分鐘後，玩家 X 的星圖才顯示 A01 的模糊熱力色塊。

## 情境二：防逆向工程與隱私邊界 (Privacy & Anti-Tracking)
**目的**：驗證聚合演算法是否能有效隱藏個體玩家的絕對座標，避免淪為「全圖開掛工具」。

- **前置條件**：
  1. 玩家 Z（探勘船）在星系 B02 的特定隱密座標 `(x: 100, y: -200, z: 500)` 採礦。
  2. 玩家 Z 的活動產生了資源採集資料，並被網路上傳。
  3. 海盜玩家 P 擁有 Premium Tier 權限，正試圖尋找獵物。
- **前置條件實作 (Data Sourcing)**：
  - **讀取來源**：玩家透過 Hub dApp 提交附帶精確座標的採集情報。
  - **技術對接**：後端預先透過 EVE EYES (`moduleName=gate` 或 `network_node`) 爬取並建立全星系座標字典檔（`eve-systems.ts`）。當收到玩家的精確座標情報時，後端呼叫 `makeCellKey()` 演算法，將其轉換為模糊且無法反推的 `GridCell`（網格區塊）再發布到星圖，保證玩家 Z 的絕對安全。
- **測試步驟**：
  1. 海盜 P 查看星系 B02 的資源熱力分佈。
  2. 海盜 P 嘗試透過 API 攔截或地圖縮放，獲取 Z 的精確座標。
- **預期結果**：
  1. 平台對資料進行 K-anonymity 或空間網格化（Grid-based fuzzing）處理。
  2. 海盜 P 只能看到「星系 B02 的第三象限有異常採礦活動」，獲得半徑範圍（例如 10,000 km）的熱力區塊。
  3. 海盜 P 必須親自前往該熱力區塊，使用艦船的定向掃描（D-Scan）才能找出 Z 的確切位置。機制保留了獵人尋找獵物的遊戲性。

## 情境三：情報懸賞與權限控制 (Bounty Intel & Access Control)
**目的**：測試玩家能否安全地將高價值情報出售，且僅限購買者或同部落成員可見。

- **前置條件**：
  1. 探勘者 M 掃描到一個未開發的高級礦帶，將其標記並上鏈。
  2. M 決定不公開，並在 Hub 上掛出「星系 C03 暗礦座標定價 10 SUI」。
  3. 買家 N 支付了 10 SUI。
- **前置條件實作 (Data Sourcing)**：
  - **讀取來源**：玩家上傳情報時附加部落 (Tribe) 存取權限；同時系統需取得雙方的身份資料。
  - **技術對接**：呼叫 EVE EYES API (`moduleName=character`) 豐潤雙方的角色名稱與 `tribe_id`。暗礦精確座標將被合約控制或加密存放；直到智能合約發出 `IntelPurchasedEvent` 且支付 / Tribe 權限驗證通過後，後端才解密並將真實座標透過 API 回傳給買家 N。
- **測試步驟**：
  1. 買家 N 支付智能合約後，查看該情報。
  2. 未付費的玩家 Q 嘗試在 Hub 上查看星系 C03。
  3. 探勘者 M 將另一筆情報的權限設定為「僅限所屬部落 (Tribe)」。
- **預期結果**：
  1. 買家 N 成功解鎖該礦帶的精確座標。
  2. 玩家 Q 在星系 C03 看不到任何特殊標記（或只看到此處有待售情報）。
  3. 只有與 M 同屬一個 Tribe 的成員能看到第二筆情報，其他人無權限調閱。

## 情境四：偽造情報與經濟嚇阻 (Intel Forgery & Economic Deterrence)
**目的**：驗證系統在面對玩家惡意上傳假情報（製造假熱力）時的防禦與懲罰機制。

- **前置條件**：
  1. 惡意玩家 F 試圖用腳本在星系 D04 狂刷「發現超級礦帶」的假情報，企圖騙人購買。
  2. 系統要求提交情報需附帶微量押金（Deposit），或與 POD 密碼學證明交叉比對。
- **前置條件實作 (Data Sourcing)**：
  - **讀取來源**：官方 World API 的跳躍證明 (Jump POD) 與 EVE EYES 的交易分析。
  - **技術對接**：
    1. **L3 密碼學證明**：要求情報附帶 `GET /v2/characters/me/jumps/{id}?format=pod`。後端用 `podPublicSigningKey` 驗證簽章與抵達星系是否吻合。
    2. **交易模式分析 (Anti-Sybil)**：呼叫 EVE EYES (`GET /api/indexer/transaction-blocks`) 審查惡意玩家 F 的鏈上發送紀錄。若發現其為「純發送情報、無正常遊戲 Move Call」的分身帳號，則大幅調降其信譽分數 (Reputation)。
- **測試步驟**：
  1. 玩家 F 發送 100 筆假情報上鏈。
  2. 其他玩家購買後發現是假的，按下「回報不實」。
- **預期結果**：
  1. 系統透過 Killmail 或 Jump POD 無法驗證 F 的活動軌跡（L2/L3 驗證失敗）。
  2. 玩家回報超過閾值後，F 質押的 SUI 被沒收並退還給受害者。
  3. F 在 Hub 上的信譽分數 (Reputation) 歸零，其未來發布的情報預設被隱藏或標記為高風險。
